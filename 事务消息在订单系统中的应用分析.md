# 事务消息在订单系统中的应用分析

## 一、业务背景

### 1.1 问题场景

在在线教育平台的订单系统中，下单操作涉及多个服务的协同工作：

```
┌─────────────────────────────────────────────────────────────────┐
│                        下单业务流程                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  用户下单                                                       │
│     │                                                           │
│     ▼                                                           │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │  ① 保存课程订单（Order 服务）                               │ │
│  │  ② 保存订单明细（Order 服务）                               │ │
│  │  ③ 保存支付订单（Pay 服务）                                 │ │
│  │  ④ 发送延时消息（取消超时订单）                            │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 核心问题

**如何保证多个服务间的数据一致性？**

- 课程订单保存成功，支付订单保存失败怎么办？
- 本地事务和消息发送如何保证原子性？
- 如何避免重复下单、重复支付？

### 1.3 解决方案对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| 本地事务 | 简单、强一致 | 无法跨服务 | 单服务内操作 |
| 2PC/XA | 强一致 | 性能差、阻塞 | 对一致性要求极高的金融场景 |
| TCC | 最终一致、可控 | 开发成本高 | 需要自定义补偿逻辑 |
| **事务消息** | 最终一致、解耦、性能好 | 需要 MQ 支持 | **订单 - 支付场景** |
| 最大努力通知 | 简单 | 一致性弱 | 通知类场景 |

## 二、事务消息原理

### 2.1 RocketMQ 事务消息机制

```
┌──────────────────────────────────────────────────────────────────────────┐
│                      RocketMQ 事务消息流程                                │
├──────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│   Producer           MQ Broker           MQ Server           Consumer    │
│      │                   │                    │                 │        │
│      │──①发送 Half 消息──>│                    │                 │        │
│      │   (Half Message)  │                    │                 │        │
│      │                   │                    │                 │        │
│      │<──②ACK 确认────────│                    │                 │        │
│      │                   │                    │                 │        │
│      │──③执行本地事务──> │                    │                 │        │
│      │   (Local Tx)      │                    │                 │        │
│      │                   │                    │                 │        │
│      │──④提交/回滚─────> │                    │                 │        │
│      │   (Commit/Rollback)                    │                 │        │
│      │                   │                    │                 │        │
│      │                   │  ⑤消息投递─────────────────────────>│        │
│      │                   │   (Consumer 消费)                     │        │
│      │                   │                    │                 │        │
│      │                   │  ⑥事务回查─────────────────> │        │
│      │<──────────────────┴────────────────────┘                 │        │
│      │   (Check Local Tx)                                       │        │
│                                                                          │
└──────────────────────────────────────────────────────────────────────────┘
```

### 2.2 三阶段提交

**阶段一：发送 Half 消息**
- 生产者发送一条"半消息"到 MQ
- MQ 存储消息，但**不投递**给消费者
- 返回 ACK 确认

**阶段二：执行本地事务**
- 生产者执行本地数据库事务
- 事务成功 → 发送 Commit
- 事务失败 → 发送 Rollback

**阶段三：消息投递**
- MQ 收到 Commit → 投递消息给消费者
- MQ 收到 Rollback → 删除消息
- MQ 超时未收到响应 → 主动回查本地事务状态

### 3.2 核心组件

#### 3.2.1 事务监听器 (CourseOrderTransactionListener)

```java
@Component
@RocketMQTransactionListener(
    txProducerGroup = "mq_courseorder_pay_group_transaction"
)
public class CourseOrderTransactionListener implements RocketMQLocalTransactionListener {

    @Autowired
    private ICourseOrderService courseOrderService;

    @Autowired
    private ICourseOrderItemService courseOrderItemService;

    /**
     * 执行本地事务 - 保存课程订单和订单明细
     * @param message MQ 消息
     * @param args 扩展参数（CourseOrderDTO）
     * @return 事务状态
     */
    @Override
    public RocketMQLocalTransactionState executeLocalTransaction(
            Message message, Object args) {

        try {
            // 1. 获取扩展参数（订单数据）
            CourseOrderDTO courseOrderDTO = (CourseOrderDTO) args;

            // 2. 保存主订单
            courseOrderService.insert(courseOrderDTO.getCourseOrder());

            // 3. 批量保存订单明细
            courseOrderItemService.insertBatch(
                courseOrderDTO.getCourseOrderItemList()
            );

            // 4. 返回提交状态
            return RocketMQLocalTransactionState.COMMIT;

        } catch (Exception e) {
            log.error("保存订单异常", e);
            return RocketMQLocalTransactionState.ROLLBACK;
        }
    }

    /**
     * 检查本地事务状态 - MQ 主动回查
     * @param message MQ 消息
     * @return 事务状态
     */
    @Override
    public RocketMQLocalTransactionState checkLocalTransaction(Message message) {
        try {
            // 1. 解析消息（支付订单数据）
            byte[] bytes = (byte[]) message.getPayload();
            String orderJSON = new String(bytes, StandardCharsets.UTF_8);
            PayOrder payOrder = JSONObject.parseObject(orderJSON, PayOrder.class);

            // 2. 根据订单号查询订单
            Wrapper<CourseOrder> wrapper = new EntityWrapper<>();
            wrapper.eq("order_no", payOrder.getOrderNo());
            CourseOrder order = courseOrderService.selectOne(wrapper);

            // 3. 订单存在则提交，否则回滚
            if (order != null) {
                return RocketMQLocalTransactionState.COMMIT;
            }
            return RocketMQLocalTransactionState.ROLLBACK;

        } catch (Exception e) {
            log.error("事务回查异常", e);
            return RocketMQLocalTransactionState.ROLLBACK;
        }
    }
}
```

#### 3.2.2 订单服务发送事务消息

```java
@Service
public class CourseOrderServiceImpl extends ServiceImpl<CourseOrderMapper, CourseOrder>
    implements ICourseOrderService {

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    /**
     * 下单（使用事务消息版本）
     */
    @Transactional(rollbackFor = Exception.class)
    public String placeOrder2(PlaceOrderDTO placeOrderDTO) {

        // ========== 1. 参数校验 ==========
        List<Long> courseIds = placeOrderDTO.getCourseIds();
        if (courseIds == null || courseIds.isEmpty()) {
            throw new GlobalBusinessException("课程 id 不能为空");
        }

        // 防重 token 校验（防止重复提交）
        Long loginId = LoginContext.getLogin().getId();
        String key = String.format(RedisConstants.REDIS_REPEAT_REFRESH_KEY_PREFIX,
                                   loginId, courseIds.get(0));
        Object orderToken = redisTemplate.opsForValue().get(key);
        if (orderToken == null || !orderToken.toString().equals(placeOrderDTO.getToken())) {
            throw new GlobalBusinessException("token 不合法，请勿重复提交");
        }

        // ========== 2. 准备订单数据 ==========
        // 2.1 查询课程信息
        JSONResult courseInfo = courseFeignApi.getCourseInfo(courseIds);
        CourseOrderDTO courseOrderDTO = createCourseOrders(courseInfo, loginId, payType);

        // 2.2 创建支付订单
        PayOrder payOrder = createPayOrder(courseOrderDTO, payType);

        // ========== 3. 发送事务消息 ==========
        TransactionSendResult result = rocketMQTemplate.sendMessageInTransaction(
            // 事务监听器组名
            RocketMQConstants.MQ_COURSEORDER_PAY_GROUP_TRANSACTION,
            // 主题：标签
            RocketMQConstants.MQ_TOPIC_ORDER + ":" + RocketMQConstants.MQ_TAGS_COURSEORDER_PAYORDER,
            // 消息体（用于生成支付单）
            MessageBuilder.withPayload(JSONObject.toJSONString(payOrder)).build(),
            // 扩展参数（用于保存课程订单）
            courseOrderDTO
        );

        // 4. 检查发送结果
        if (result.getLocalTransactionState() != LocalTransactionState.COMMIT_MESSAGE) {
            throw new GlobalBusinessException("订单生成失败，请稍后重试");
        }

        // ========== 5. 发送延迟消息（超时取消）==========
        rocketMQTemplate.syncSend(
            RocketMQConstants.MQ_TOPIC_COURSEORDER_CANCEL_DELAY + ":" +
            RocketMQConstants.MQ_TAGS_COURSEORDER_CANCEL_DELAY,
            MessageBuilder.withPayload(payOrder.getOrderNo()).build(),
            3000,  // 超时时间
            3      // 延迟等级（3 级 = 10 秒后消费）
        );

        // 6. 删除防重 token
        redisTemplate.delete(key);

        return courseOrderDTO.getCourseOrder().getOrderNo();
    }
}
```

#### 3.2.3 支付服务消费消息

```java
@Component
@Slf4j
public class PayOrderConsumer {

    @Autowired
    private IPayOrderService payOrderService;

    /**
     * 消费订单消息，保存支付订单
     */
    @RocketMQMessageListener(
        topic = "mq_topic_order",
        consumerGroup = "pay_order_consumer_group",
        selectorExpression = "mq_tags_courseorder_payorder"
    )
    public class PayOrderMessageListener implements RocketMQListener<String> {

        @Override
        public void onMessage(String message) {
            try {
                // 1. 解析支付订单数据
                PayOrder payOrder = JSONObject.parseObject(message, PayOrder.class);

                // 2. 保存支付订单
                payOrderService.insert(payOrder);

                log.info("支付订单保存成功，订单号：{}", payOrder.getOrderNo());

            } catch (Exception e) {
                log.error("保存支付订单失败", e);
                // 抛出异常，MQ 会重试
                throw new RuntimeException("保存支付订单失败", e);
            }
        }
    }
}
```

## 五、关键设计点

### 5.1 防重复提交

**问题：** 用户快速点击多次，导致重复下单。

**解决方案：** Redis Token 机制

```java
// 1. 下单前生成 token，存入 Redis
String token = UUID.randomUUID().toString();
redisTemplate.set(key, token, 300); // 5 分钟过期

// 2. 提交时校验并删除 token
Object orderToken = redisTemplate.opsForValue().get(key);
if (orderToken == null || !orderToken.toString().equals(token)) {
    throw new GlobalBusinessException("请勿重复提交");
}
redisTemplate.delete(key);
```

### 5.2 延迟消息（超时取消）

**问题：** 用户下单后未支付，需要超时取消订单释放库存。

**解决方案：** RocketMQ 延迟消息

```java
// 发送延迟消息，3 级延迟 = 10 秒后消费
rocketMQTemplate.syncSend(
    "mq_topic_courseorder_cancel_delay:mq_tags_courseorder_cancel_delay",
    MessageBuilder.withPayload(orderNo).build(),
    3000,  // 超时时间
    3      // 延迟等级
);
```

**延迟等级说明：**
| 等级 | 延迟时间 |
|------|----------|
| 1 | 1 秒 |
| 2 | 5 秒 |
| 3 | 10 秒 |
| 4 | 30 秒 |
| 5 | 1 分钟 |
| ... | ... |
| 18 | 2 小时 |

### 5.3 事务回查机制

**问题：** 如果 Commit 消息丢失，MQ 如何处理 Half 消息？

**解决方案：** MQ 主动回查本地事务状态

```java
@Override
public RocketMQLocalTransactionState checkLocalTransaction(Message message) {
    // 1. 解析消息，获取订单号
    PayOrder payOrder = JSONObject.parseObject(...);

    // 2. 查询本地数据库
    CourseOrder order = courseOrderService.selectByOrderNo(payOrder.getOrderNo());

    // 3. 订单存在则提交，否则回滚
    if (order != null) {
        return RocketMQLocalTransactionState.COMMIT;
    }
    return RocketMQLocalTransactionState.ROLLBACK;
}
```

## 七、异常处理

### 7.1 异常场景处理

| 场景 | 处理方案 |
|------|----------|
| Half 消息发送失败 | 直接抛出异常，返回用户下单失败 |
| 本地事务执行失败 | 返回 ROLLBACK，MQ 删除 Half 消息 |
| Commit 消息丢失 | MQ 主动回查，根据本地事务状态决定 |
| 消费者处理失败 | MQ 重试投递（最多 16 次） |
| 延迟消息发送失败 | 记录日志，定时任务补偿扫描 |

### 7.2 重试机制

```java
@RocketMQMessageListener(
    topic = "mq_topic_order",
    consumerGroup = "pay_order_consumer_group",
    maxReconsumeTimes = 5  // 最多重试 5 次
)
```

## 八、总结

### 8.1 事务消息优势

| 优势 | 说明 |
|------|------|
| **解耦** | Order 服务和 Pay 服务独立部署，通过 MQ 通信 |
| **最终一致性** | 保证订单和支付单数据最终一致 |
| **高性能** | 不阻塞主流程，异步处理 |
| **可靠性** | 事务回查机制保证消息不丢失 |

### 8.2 适用场景

- 订单 - 支付流程
- 注册 - 发送欢迎邮件
- 充值 - 发放积分
- 任何需要**跨服务数据一致性**的场景

### 8.3 注意事项

1. **幂等性**：消费者必须实现幂等处理（可能重复消费）
2. **事务回查**：必须实现 checkLocalTransaction 方法
3. **本地事务**：executeLocalTransaction 中必须开启本地事务
4. **消息格式**：消息体尽量精简，复杂数据通过 ID 查询
