# RocketMQ 延迟消息在订单超时取消中的应用分析

## 一、应用场景

在在线教育平台的订单系统中，用户下单后需要在**30 分钟内完成支付**，超时未支付的订单需要自动取消。这是一个典型的**延时任务**场景。

### 1.1 业务需求

- 用户下单成功后开始计时
- 30 分钟内未支付则自动取消订单
- 取消订单的同时需要：
  - 更新课程订单状态为"自动取消"
  - 更新支付订单状态为"已取消"
  - 释放已占用的库存

### 1.2 传统方案 vs 延迟消息方案

| 方案 | 实现方式 | 缺点 | 优点 |
|------|----------|------|------|
| 定时任务轮询 | 每分钟扫描数据库查询超时订单 | 实时性差、数据库压力大 | 实现简单 |
| JDK DelayQueue | 内存队列存储待处理订单 | 服务重启数据丢失、无法集群 | 实时性好 |
| Redis ZSet | 使用分数存储过期时间 | 需要主动轮询、实现复杂 | 实时性较好 |
| **RocketMQ 延迟消息** | 发送延迟消息，到期自动消费 | 依赖 MQ 中间件 | 实时性好、可靠、支持集群 |

## 二、RocketMQ 延迟消息原理

### 2.1 延迟等级

RocketMQ 提供了固定的延迟等级，不支持任意延迟时间：

| 等级 | 延迟时间 |
|------|----------|
| 1 | 1 秒 |
| 2 | 5 秒 |
| 3 | 10 秒 |
| 4 | 30 秒 |
| 5 | 1 分钟 |
| 6 | 2 分钟 |
| 7 | 3 分钟 |
| 8 | 4 分钟 |
| 9 | 5 分钟 |
| 10 | 6 分钟 |
| 11 | 7 分钟 |
| 12 | 8 分钟 |
| 13 | 9 分钟 |
| 14 | 10 分钟 |
| 15 | 20 分钟 |
| 16 | 30 分钟 |
| 17 | 1 小时 |
| 18 | 2 小时 |

## 三、项目实现方案

### 3.1 常量定义

在 `RocketMQConstants.java` 中定义延迟消息相关的主题和标签：

```
主题常量：mq_topic_courseorder_cancel_delay
标签常量：mq_tags_courseorder_cancel_delay
```

### 3.2 发送端实现

**位置**：订单服务 - `CourseOrderServiceImpl.java`

**发送时机**：订单创建成功后立即发送延迟消息

**消息内容**：订单号（String 类型）

**延迟等级**：3 级（10 秒）或 16 级（30 分钟）

#### 普通订单创建流程

```
┌────────────────────────────────────────────────────────────────┐
│                    订单创建 + 延迟消息发送流程                    │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  ① 用户发起下单请求                                              │
│         │                                                       │
│         ▼                                                       │
│  ② 创建课程订单和订单明细                                        │
│         │                                                       │
│         ▼                                                       │
│  ③ 创建支付订单                                                  │
│         │                                                       │
│         ▼                                                       │
│  ④ 发送事务消息（保证订单和支付单数据一致性）                     │
│         │                                                       │
│         ▼                                                       │
│  ⑤ 发送延迟消息（30 分钟后取消订单）                              │
│         │                                                       │
│         ▼                                                       │
│  ⑥ 返回订单号，下单完成                                          │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

**核心代码逻辑**：

1. 构建消息体：订单号字符串
2. 调用 `rocketMQTemplate.syncSend()` 发送
3. 参数说明：
   - 主题 + 标签：`mq_topic_courseorder_cancel_delay:mq_tags_courseorder_cancel_delay`
   - 消息体：`MessageBuilder.withPayload(订单号).build()`
   - 超时时间：3000ms
   - 延迟等级：3（10 秒）或 16（30 分钟）
4. 检查发送状态，失败则抛出异常

### 3.3 消费端实现

项目中有两个消费者监听同一延迟消息，分别处理不同的业务逻辑：

#### 消费者 1：课程订单取消服务

**位置**：`CancelCourseOrderService.java`

**消费者组**：`service-order-delay-consumer`

**消费逻辑**：
1. 解析消息体获取订单号
2. 构建订单状态更新 DTO
3. 调用服务更新订单状态为"自动取消"（AutoCancel）
4. 记录日志

#### 消费者 2：支付订单取消服务

**位置**：`CancelPayOrderService.java`

**消费者组**：`service-pay-delay-consumer`

**消费逻辑**：
1. 解析消息体获取订单号
2. 调用支付服务取消支付订单
3. 调用支付宝接口关闭支付宝订单（TODO）
4. 记录日志

### 3.4 广播模式

两个消费者都配置为**广播模式**（`MessageModel.BROADCASTING`）：

```
同一消息会被所有消费者组的每个实例消费
```

**为什么使用广播模式**：

- 课程订单服务和支付服务是两个独立的服务
- 同一延迟消息需要同时通知两个服务
- 广播模式确保每个服务都能收到消息并执行各自的取消逻辑

### 5.2 延迟等级选择

项目中使用的延迟等级为 **3 级（10 秒）**，实际生产环境应使用 **16 级（30 分钟）**。

**修改建议**：
```
当前配置：delayLevel = 3  （10 秒，用于测试）
生产配置：delayLevel = 16 （30 分钟，符合业务需求）
```

## 六、异常处理与兜底策略

### 6.1 延迟消息发送失败

**异常处理**：
- 捕获异常并记录日志
- 抛出业务异常，提示用户"延迟消息发送失败"

**兜底策略**：
1. **重试发送**：可以配置重试次数
2. **数据库记录**：记录延迟消息发送日志，定时任务扫描补偿
3. **告警通知**：发送通知给运维人员

### 6.2 消息消费失败

**RocketMQ 重试机制**：
- 默认重试 16 次
- 重试间隔逐渐增加
- 重试失败后进入死信队列

**项目处理**：
- 使用 `@SneakyThrows` 抛出异常，让 MQ 自动重试
- 消费日志记录，便于排查问题

## 七、优势总结

| 优势 | 说明 |
|------|------|
| **解耦** | 订单服务和支付服务通过 MQ 解耦，互不依赖 |
| **可靠** | MQ 保证消息不丢失，支持重试 |
| **实时** | 到期立即消费，实时取消订单 |
| **集群** | 支持多实例部署，提高处理能力 |
| **扩展** | 新增消费者只需监听同一主题即可 |

## 八、注意事项

### 8.1 延迟等级限制

- RocketMQ 只支持固定的 18 个延迟等级
- 不能设置任意延迟时间
- 需要根据业务需求选择合适的等级

### 8.2 消息幂等性

- 消费者可能重复消费同一消息
- 取消订单操作需要保证幂等性
- 可以通过订单状态判断是否已取消

### 8.3 广播模式特性

- 广播模式下，每个消费者实例都会收到消息
- 不支持负载均衡
- 适合需要多方同时处理的场景

### 8.4 消息持久化

- 延迟消息存储在 Broker 的 SCHEDULE_TOPIC 中
- 服务重启不影响延迟消息
- Broker 宕机时消息可能丢失（依赖 Broker 配置）

## 九、总结

本项目使用 RocketMQ 延迟消息实现了订单超时自动取消功能，主要特点包括：

1. **架构清晰**：发送端和消费端职责明确，易于维护
2. **解耦服务**：订单服务和支付服务通过 MQ 解耦
3. **可靠保证**：MQ 保证消息可靠投递，支持重试
4. **扩展性强**：新增业务只需添加新的消费者

该方案是分布式系统中处理延时任务的推荐方案，适合在毕业设计中作为技术亮点展示。
